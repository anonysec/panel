# Requirements Document

## Introduction

Upgrade the KorisPanel backup system from the current basic JSON export to a comprehensive SQL dump-based backup/restore system. The new system creates compressed archives containing a full MariaDB SQL dump plus VPN configuration files collected from nodes, provides downloadable backups from the admin panel, supports scheduled automatic backups with retention policies, and enables full restore from uploaded backup files.

## Glossary

- **Panel**: The central KorisPanel web application that manages nodes, customers, and VPN services
- **Backup_Service**: The Go backend module responsible for creating, managing, scheduling, and restoring backups
- **Backup_Archive**: A compressed .tar.gz file containing a SQL dump, node configuration files, and a metadata manifest
- **SQL_Dump**: A complete MariaDB database dump generated by the mysqldump utility
- **Config_Collection**: The process of gathering VPN configuration files from nodes via the Task_System
- **Task_System**: The existing mechanism where the Panel dispatches tasks to Node_Agent via /api/node/tasks/poll
- **Node_Agent**: The lightweight agent process deployed on each VPN server node that executes tasks and reports status
- **Backup_Metadata**: A JSON manifest file within the Backup_Archive describing timestamp, size, contents inventory, and checksum
- **Retention_Policy**: A configurable rule that determines how many backups to keep and when to delete older backups
- **Backup_Schedule**: A cron-like configuration defining when automatic backups are created (daily, weekly, or custom)
- **Admin_UI**: The Vue 3 admin dashboard SPA for managing all panel resources
- **Storage_Directory**: The local filesystem path where backups are stored (/opt/KorisPanel/backups/)
- **Checksum**: A SHA-256 hash computed over the Backup_Archive for integrity verification

## Requirements

### Requirement 1: Full SQL Dump Backup Creation

**User Story:** As an administrator, I want to create a full MariaDB SQL dump backup, so that I have a complete database snapshot that can be fully restored.

#### Acceptance Criteria

1. WHEN an administrator triggers a manual backup, THE Backup_Service SHALL execute mysqldump with the --single-transaction and --routines flags against the configured database
2. THE Backup_Service SHALL use the MYSQL_PWD environment variable for password passing to prevent credential exposure in the process list
3. THE Backup_Service SHALL extract MySQL credentials from the PANEL_DB_DSN environment variable using the existing mysqlCredsFromDSN function
4. IF mysqldump returns a non-zero exit code, THEN THE Backup_Service SHALL record the backup status as "failed" with the stderr output as the error message
5. THE Backup_Service SHALL generate a SQL dump that includes all tables, views, triggers, and routines in the database

### Requirement 2: Node Configuration File Collection

**User Story:** As an administrator, I want backups to include VPN configuration files from all nodes, so that I can restore the complete system state including node-specific settings.

#### Acceptance Criteria

1. WHEN a backup is initiated, THE Backup_Service SHALL dispatch a "backup.collect_configs" task to each online Node_Agent
2. WHEN the Node_Agent receives a "backup.collect_configs" task, THE Node_Agent SHALL collect files from /etc/openvpn/, /etc/wireguard/, /etc/ipsec.d/, and /etc/xl2tpd/ directories
3. WHEN the Node_Agent completes config collection, THE Node_Agent SHALL return the collected files as a base64-encoded tar archive in the task completion response
4. THE Backup_Service SHALL store collected node configs in the Backup_Archive under a directory structure of configs/{node_name}/{path}
5. IF a Node_Agent does not respond within 60 seconds, THEN THE Backup_Service SHALL proceed with the backup without that node's configs and record the node as "skipped" in Backup_Metadata
6. IF a Node_Agent is offline at backup time, THEN THE Backup_Service SHALL skip that node and record the node as "offline" in Backup_Metadata

### Requirement 3: Compressed Archive Packaging

**User Story:** As an administrator, I want the backup packaged as a single compressed archive, so that it is easy to transfer, store, and manage.

#### Acceptance Criteria

1. THE Backup_Service SHALL package the SQL_Dump and collected node configs into a single .tar.gz Backup_Archive
2. THE Backup_Service SHALL name the archive file using the format "backup-{YYYY-MM-DD-HHmmss}.tar.gz"
3. THE Backup_Archive SHALL contain the following structure: dump.sql at the root, configs/ directory with node subdirectories, and manifest.json at the root
4. THE Backup_Service SHALL store the completed Backup_Archive in the Storage_Directory (/opt/KorisPanel/backups/)
5. THE Backup_Service SHALL create the Storage_Directory with permissions 0750 if the directory does not exist

### Requirement 4: Backup Metadata and Integrity Verification

**User Story:** As an administrator, I want each backup to include metadata and a checksum, so that I can verify backup integrity before restoring.

#### Acceptance Criteria

1. THE Backup_Service SHALL generate a manifest.json file containing: timestamp (ISO 8601), database name, archive size in bytes, list of included node names, list of skipped nodes with reasons, and the panel version
2. THE Backup_Service SHALL compute a SHA-256 Checksum over the completed Backup_Archive file
3. THE Backup_Service SHALL store the Checksum in the database backup record and in a companion .sha256 file alongside the archive
4. WHEN an administrator requests integrity verification, THE Backup_Service SHALL recompute the SHA-256 hash of the archive file and compare the computed hash with the stored Checksum
5. IF the computed hash does not match the stored Checksum, THEN THE Backup_Service SHALL report the backup as "corrupted"

### Requirement 5: Backup Download from Admin Panel

**User Story:** As an administrator, I want to download backup files from the admin panel, so that I can store copies off-site or transfer them to another server.

#### Acceptance Criteria

1. WHEN an administrator requests a backup download, THE Panel SHALL serve the Backup_Archive file with Content-Type "application/gzip" and Content-Disposition header for download
2. THE Panel SHALL stream the file from the Storage_Directory without loading the entire file into memory
3. IF the requested backup file does not exist on disk, THEN THE Panel SHALL return a 404 response with error "backup_file_not_found"

### Requirement 6: Scheduled Automatic Backups

**User Story:** As an administrator, I want to configure automatic backup schedules, so that backups are created regularly without manual intervention.

#### Acceptance Criteria

1. THE Admin_UI SHALL provide schedule configuration options: daily (with configurable hour), weekly (with configurable day and hour), or disabled
2. THE Backup_Service SHALL evaluate the schedule every minute and trigger a backup when the current time matches the configured schedule
3. THE Backup_Service SHALL store the schedule configuration in the panel_settings table with key "backup_schedule"
4. THE Backup_Service SHALL prevent concurrent backup execution by checking if another backup is currently in progress
5. WHEN a scheduled backup completes, THE Backup_Service SHALL record the completion time and status in the database

### Requirement 7: Backup Retention Policy

**User Story:** As an administrator, I want to configure how many backups to keep, so that disk space is managed automatically without manual cleanup.

#### Acceptance Criteria

1. THE Admin_UI SHALL provide a retention configuration field specifying the maximum number of backups to keep (default: 7)
2. WHEN a new backup completes successfully, THE Backup_Service SHALL count existing backups and delete the oldest archives that exceed the retention limit
3. THE Backup_Service SHALL delete both the archive file and its companion .sha256 file when removing old backups
4. THE Backup_Service SHALL store the retention configuration in the panel_settings table with key "backup_retention_count"
5. THE Backup_Service SHALL not delete backups that are currently being downloaded or restored

### Requirement 8: Restore from Backup Upload

**User Story:** As an administrator, I want to restore the system from a backup file, so that I can recover from data loss or migrate to a new server.

#### Acceptance Criteria

1. WHEN an administrator uploads a backup file for restore, THE Backup_Service SHALL verify the file is a valid .tar.gz archive containing dump.sql and manifest.json
2. WHEN restoring, THE Backup_Service SHALL apply the SQL_Dump to the database using the mysql client command with --force flag
3. WHEN restoring, THE Backup_Service SHALL dispatch "backup.restore_configs" tasks to online nodes with their respective config archives from the configs/ directory
4. WHEN the Node_Agent receives a "backup.restore_configs" task, THE Node_Agent SHALL extract the provided config files to their original paths and restart affected VPN services
5. IF the uploaded file fails validation, THEN THE Backup_Service SHALL return an error describing the validation failure without modifying the database
6. THE Backup_Service SHALL verify the uploaded file checksum against the manifest before proceeding with restore
7. THE Backup_Service SHALL create a pre-restore backup automatically before applying the restore operation

### Requirement 9: Admin UI Backup Management

**User Story:** As an administrator, I want a backup management page in the admin panel, so that I can view, download, restore, schedule, and delete backups from a single interface.

#### Acceptance Criteria

1. THE Admin_UI SHALL display a backup list showing: filename, creation timestamp, file size, status (completed/failed/in_progress), and included nodes count
2. THE Admin_UI SHALL provide action buttons per backup: download, restore, verify integrity, and delete
3. THE Admin_UI SHALL display the current schedule and retention policy settings with edit capability
4. THE Admin_UI SHALL provide a "Create Backup Now" button to trigger manual backup creation
5. WHEN a backup or restore is in progress, THE Admin_UI SHALL display a progress indicator showing the current step (dumping database, collecting configs, packaging archive)
6. THE Admin_UI SHALL display a confirmation dialog before restore operations warning that current data will be overwritten

### Requirement 10: Backup Status Tracking

**User Story:** As an administrator, I want to track the status of backup operations, so that I can verify backups completed successfully and diagnose failures.

#### Acceptance Criteria

1. THE Backup_Service SHALL store backup records in a backups table with columns: id, filename, status (in_progress/completed/failed), size_bytes, checksum, nodes_included, nodes_skipped, started_at, completed_at, error_message
2. WHEN a backup starts, THE Backup_Service SHALL create a record with status "in_progress"
3. WHEN a backup completes, THE Backup_Service SHALL update the record with status "completed", final size, and checksum
4. IF a backup fails at any stage, THEN THE Backup_Service SHALL update the record with status "failed" and the error message
5. THE Backup_Service SHALL retain backup records in the database even after the archive file is deleted by retention policy

### Requirement 11: Resource Constraints Compliance

**User Story:** As a system operator, I want the backup system to operate within the resource limits of a 1GB RAM single-core VPS, so that the panel remains responsive during backup operations.

#### Acceptance Criteria

1. THE Backup_Service SHALL stream the mysqldump output directly to the archive writer without buffering the entire dump in memory
2. THE Backup_Service SHALL limit concurrent backup operations to one at a time
3. THE Backup_Service SHALL use streaming compression (gzip writer) to avoid holding uncompressed data in memory
4. WHEN processing node config responses, THE Backup_Service SHALL decode and write base64 data in chunks not exceeding 4MB per node
